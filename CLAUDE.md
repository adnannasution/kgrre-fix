# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

KGRRE ("Kilang Graph") is a **knowledge-graph viewer for refinery equipment**. It consumes the output of an **external ETL pipeline** (CSV/JSON files, produced elsewhere — e.g. a Colab notebook) and renders it as an interactive graph with Equipment 360, Refinery Unit dashboards, schema/depth explorers, and audit review. **The app never rebuilds the graph from raw Excel/domain data** — it only ingests the ETL's already-shaped output. UI strings are in Bahasa Indonesia.

It deploys to **Railway** (Dockerfile + PostgreSQL plugin). Data lives in **PostgreSQL** (one shared database; datasets isolated by `dataset_id` + Row-Level Security). Set `DATABASE_URL` via `.env` for local dev (use Railway's `DATABASE_PUBLIC_URL`) or Railway injects it in production.

## Commands

The Python venv is at `.venv/`. The `.mjs` scripts auto-pick `.venv/Scripts/python.exe` (Windows) vs `.venv/bin/python` (POSIX). **Requires `DATABASE_URL`** (in `.env`) for anything that touches the DB.

- **Dev (both servers):** `pnpm dev` — Uvicorn (`backend.app:app`) on `127.0.0.1:8765` with `--reload` + Vite on `:5173`. Vite proxies `/api` → `:8765`; use the Vite URL during development.
- **Build:** `pnpm build` — `tsc -b && vite build` (frontend → `dist/`).
- **Production/single-server:** `pnpm start` — Uvicorn binds `0.0.0.0:$PORT` (fallback 8765); FastAPI serves built `dist/` as static files, so the whole app is one origin.
- **Tests:** `pnpm test` — `tsc -b` then Python `unittest discover -s tests`. (DB-touching code isn't unit-tested; tests are pure-Python helpers + scanner.)
- **Single test:** `.venv/Scripts/python.exe -m unittest tests.test_property_query` (set `PYTHONPATH=.`).

Python deps in `requirements.txt` (psycopg3, fastapi, uvicorn, python-dotenv — no DuckDB); frontend deps in `package.json`. Deploy via `Dockerfile` (multi-stage: build frontend, run uvicorn) + `railway.json`.

## Architecture

Three-layer split: React frontend → FastAPI backend → **PostgreSQL** (one shared DB; per-dataset isolation via `dataset_id` + RLS).

### Backend (`backend/`)
- **`app.py`** — the entire FastAPI surface (~34 routes, all under `/api/datasets/{id}/...`). `db_for(dataset_id)` returns a `_ScopedConnection`: it borrows a pooled connection and runs `set_config('app.dataset_id', …)` so **Row-Level Security auto-scopes every query to that dataset** — there is no manual `WHERE dataset_id=…` in endpoint SQL, and cross-dataset reads are impossible. Endpoints keep the old `connection = db_for(...); try: … finally: connection.close()` shape; `.close()` returns the connection to the pool. `@app.on_event("startup")` runs `ensure_schema()` (creates tables + RLS).
- **`database.py`** — psycopg3 + `ConnectionPool` (global `dict_row` factory). `initialize()` = DDL, `finalize()` = indexes incl. a **GIN index on `properties_json` jsonb**, `enable_rls()` = RLS policies. 8 per-dataset tables (each has `dataset_id text DEFAULT current_setting('app.dataset_id')`) + a `dataset_catalog` table. Central tables: `kg_node`, `kg_relationship`. **Custom attributes live in a `properties_json` jsonb column** — queried with `properties_json ->> 'key'`. Use **`scoped(dataset_id)`** for writes/imports and **`fetch_tuple(conn, sql)`** for positional/scalar fetches (the pool's `dict_row` makes `fetchone()[0]` fail — `fetch_tuple` uses a `tuple_row` cursor).
- **`importer.py`** — async import pipeline. Each import runs on a **background thread** (`JOBS` dict + `JOBS_LOCK`) inside a `scoped(dataset_id, autocommit=True)` connection; the frontend polls `/imports/{id}`. **CSV is loaded in row-batches via `_chunked_csv_insert`** (read with `csv.reader`, build rows in Python, `cursor.executemany(INSERT … ON CONFLICT DO NOTHING)` per 5000 rows, commit each batch). `_py_bigint`/`_py_double`/`_py_jsonb` cast in Python; `_row_to_json` serializes a whole CSV row to a jsonb object (replaces SQL `to_jsonb(t)`); `_pick(header_idx, *names)` resolves flexible column names. **Do not revert to the old temp-table + `COPY … FROM STDIN` + `INSERT … SELECT … to_jsonb(t)` approach** — see "Large-data import" below. ZIP imports extract to a tempdir with path-traversal guards (max 2 path levels; second level only for `input_metadata`/`analysis_ready`).
- **`scanner.py`** — recognizes ETL output files **by filename convention** (see below) and validates required columns.
- **`config.py`** — dataset registry lives in the `dataset_catalog` **table** (`list_datasets`/`get_dataset_row`/`insert_dataset`/`rename_dataset`/`delete_dataset`). `UPLOADS_DIR` is an ephemeral tempdir for ZIP extraction only. No filesystem dataset storage.

### Frontend (`src/`)
- **`App.tsx`** — single large component holding all page routing (overview, import, executive, insight, equipment, graph, depth, review, datasets) and query state.
- **UI layout conventions (don't regress):**

Don't reintroduce hard `.slice(0, N)` caps for data the user reads (Reliability Insight, Executive coverage/quality, Equipment 360 cards all use `Paged`). It's a render-prop (not a bare hook) so hook counts stay stable when the Manager/VP view switches.
  - **Tables:** `.table-panel.fit` = no horizontal scroll (text wraps, `min-width: 0`) — used across Reliability Insight and the Data coverage / Relationship quality cards. Plain `.table-panel` keeps `min-width: 720px` + scroll for wide full-width tables (RU equipment summary, Data Review). `.two-column.balanced` = equal-width 2-column grid (collapses at ≤760px — the media-query rule must also name `.two-column.balanced` or it loses on specificity).
  - **Insight/Executive layout:** the widest table (Equipment action queue, RU reliability portfolio) sits full-width on top; the rest pair up inside `.two-column.balanced`.
  - **Graph Explorer:** a 3-column control row (search & filters | stage controls + diagnosis prompt | inspector) on top, with the graph canvas in a full-width **`.graph-viewer`** card below it (grid-column `1 / -1`). The Property Query panel is **collapsed by default** (`queryPanelOpen`); the diagnosis copy buttons are filled-blue **`.primary small`** (`Copy prompt: Engineer / Reliability Mgr / Maintenance Mgr / VP Reliability`, full label in `title`, same color design as the segmented active button) and may wrap to a second row — never clip them with `nowrap`/`overflow`. The graph legend is compact and aligned on the same top row as the zoom toolbar (`.graph-legend { top: 13px }`) so it doesn't cover nodes.

Domain cards paginate via `Paged` with the card `key` including `view.equipment.id` so pages reset on equipment change.

- **`components/GraphView.tsx`** — `react-force-graph-2d` renderer. **Node color is keyed off `node_type`**; unknown types fall back to a default color.

### Graph traversal
`neighbors` and `directed-descendants` do **breadth-first search in Python** ([app.py](backend/app.py)), not recursive SQL. Each hop runs a bounded edge query (`_frontier_edges` / `_directed_frontier_edges`) and the Python loop tracks the visited set, paths, and cycle detection. Per-query node count is capped by a `limit` param (default 150, max 1000; `refinery_unit` roots are force-capped at 150 because they are high-degree hubs).

- **Tag-matched domains are synthetic *candidate* links, shared by graph AND Equipment 360 (don't regress).** Some domains have **no real `EQUIPMENT_HAS_*` edge** to a given equipment (readiness_record / inspection never do; reliability_observation / rkap_program / issue do in general but many equipment lack a recorded edge), so they'd never appear in that equipment's neighborhood / 360. The candidate-synthesis matcher is centralized in **`_iter_tag_match_candidates(connection, root, root_properties, present_ids, budget, …)`** — a generator yielding `(config, row, matched_token)` — reused by **both** `neighbors` (builds dashed candidate edges via `_synthetic_tag_link_edge`, only when `include_candidates` and root is `equipment`) and **`equipment_360`** (appends related items flagged `is_candidate=True` / `confidence=None` / `matched_token`, gated by `include_candidates` default `True`). It reuses the *same* matcher as the prompt path: `_tag_matched_domain_rows` + `TAG_MATCH_GRAPH_LINKS` (5 domains: readiness, inspection, reliability, RKAP, issue) + `_equipment_match_tokens` (incl. `functional_location` + `/NN` suffix strip) + the `\y` RU filter + `_compact_match_condition`. **Dedupe via `present_ids` (seeded from the real related/neighbor node ids)** means a record already linked by a real edge to this equipment is never duplicated as a candidate. The whole injection is wrapped in `try/except` so a bad row can never 500 the graph or 360 (same defensive pattern as `_reliability_engineering_signals`). Adding speculative `tag_columns` is safe: a missing `properties_json ->> 'col'` → NULL → compacts to `''` → never overlaps the ≥4-char equipment tokens. Counts in `_virtual_node_properties` stay `NOT is_candidate` and are unaffected; the 360 frontend shows recorded vs candidate counts separately and badges candidate cards "kandidat".
- **The left "Search & filters → Start node" inputs (`node_type`/`refinery_unit`/`equipment_code`) are for finding the start node — they must NOT scope the graph.** The `App.tsx` graph calls (`loadGraph`/`applyFilters`) pass these **empty** to `neighbors`. Passing them collapses the neighborhood: `_node_filter` exact-matches `node_type`, so e.g. `node_type=equipment` around an equipment root drops every neighbor (maintenance orders/notifications) → 1 node, 0 edges — and it also makes the synthetic-link injection gate skip. Keep them wired to `api.search` only, never to the graph query.

## ETL output contract (critical when changing the upstream pipeline)

The importer is **schema-flexible by design** — preserve this when modifying it:
- **Required files:** `nodes.csv` (cols `node_id, node_type, label, properties_json`) and `relationships.csv` (cols `relationship_id, source_node_id, target_node_id, relationship_type`). Missing required columns → file marked `Invalid` in `scanner.py`.
- **`node_type` / `relationship_type` are free-text** (no enum). New types ingest with zero code changes; they just need a color added in `GraphView.tsx` for custom styling.
- **Extra attributes** belong inside `properties_json`, not as new columns. The importer's `col()` / `first_col()` helpers ignore unrecognized raw columns safely.
- **New file kinds are matched by name** in `scanner.py::package_file_type`:
  - `domain_*.csv` → `domain_record` table (auto, no code change)
  - files under `analysis_ready/` and `input_metadata/` → auto-ingested
  - `relationship_candidates.csv` → audit-only edges, hidden unless candidate mode is on
  - Any other new filename returns `None` and is **silently skipped** — to support it, register it in `RECOMMENDED_FILES`/`ANALYSIS_FILES`/`AUDIT_FILES` and add a handler in `importer.py`.

## Conventions

- Backend SQL placeholders are psycopg-style **`%s`** (positional). The one `?` in the codebase is a regex literal (`QUERY_RE`), not SQL.
- JSON access is **`properties_json ->> 'key'`** (jsonb); use `to_jsonb(t)` / `jsonb_build_object(...)`; numeric stats use `percentile_cont(0.5) WITHIN GROUP (ORDER BY x)`; regex match is `a ~ b`.
- **PostgreSQL regex word boundary is `\y`/`\Y`, NOT `\b`.** In Postgres ARE (the flavor used by `~`), `\b` means a *backspace* character — so a pattern like `'\bRU VI\b'` silently matches nothing and the filter returns 0 rows (no error). This bit the RU filters in the prompt path (`_ru_context_prompt_evidence`/`_readiness_context_for_node`/`_inspection_tag_match`), making readiness/inspection tag-match always empty while the Python-side dashboard (`_readiness_association_summary`, which matches RU via `_ru_key`) looked fine. Always use `\y` for word boundaries in any regex passed to `~`. (Python's `re` — e.g. `_ru_key` — does use `\b`; don't "fix" those.)
- Per-dataset isolation is enforced by **RLS** keyed on `current_setting('app.dataset_id')`, set by `_ScopedConnection`/`scoped()`. Don't bypass it: read/write per-dataset tables only through `db_for()` or `scoped()`, never a raw pool connection, or RLS won't apply.
- **Don't** make `node_type`/`relationship_type` enums or `properties_json` a rigid schema — the importer is intentionally schema-flexible (new ETL node types / properties must ingest without code changes). Keep `col()`/`first_col()`/`package_file_type()` tolerant.
- All user-visible strings (errors, messages) are Indonesian — match this.

## Large-data import (hard-won — don't regress)

The real ETL output is huge: `nodes.csv` ~990 MB (~1.5M rows), `relationships.csv` ~1.2 GB (~2.3M rows), plus 376 MB+ `domain_*.csv` — ~417 MB zipped, ~3.2 GB extracted. Getting this into Railway PostgreSQL took several iterations; the rules below encode what failed and why.

- **Browser upload won't work for the full set.** Railway's HTTP proxy times out (~5 min) and a 417 MB upload exceeds it. For large data, import **from a laptop** with `scripts/import_local.py "<zip>" "<dataset name>"` (reads `DATABASE_URL` from `.env`, reuses the real importer pipeline, runs synchronously so errors surface). It extracts to `<zip_dir>/.import_tmp/` (a dir on the *same drive as the zip*) — **not** the OS temp dir, because `C:` was nearly full (1.7 GB free) while `D:` had 58 GB. It does **not** delete the source zip (unlike `start_zip_import`, which does).
- **Why chunked insert (not temp-table + COPY).** The original importer loaded each whole CSV into a temp table via one `COPY`, then one `INSERT … SELECT … to_jsonb(t)` over all rows. Two failure modes resulted: (1) the entire import ran as **one giant transaction** (pool connections commit only on clean context exit) so the server held millions of uncommitted rows → "server closed the connection unexpectedly"; (2) even after `autocommit`, a single 990 MB `COPY`/INSERT and especially the domain `to_jsonb(t)` over a 376 MB temp table **hung** (confirmed via `pg_stat_activity`: a `COPY raw_domain_…` stuck for 21 min, client CPU flatlined). The fix in `_chunked_csv_insert` keeps every transaction tiny (5000 rows) and does casts/JSON in Python, so memory never balloons on the server.
- **`scoped(dataset_id, autocommit=True)`** is required for imports so each batch commits independently. The session var `app.dataset_id` is set with `is_local=false`, so it survives across commits and RLS/`DEFAULT dataset_id` stay correct.
- **Don't kill or terminate shared Postgres connections** (`pg_terminate_backend`) to "clean up" a hung import — that's interfering with shared infra. Kill the local Python client instead; the server reaps its orphaned backend on its own.
- **Resetting between attempts:** `scripts/_reset_db.py` `TRUNCATE`s all data tables + `dataset_catalog` (keeps schema/RLS/volume). Prefer this over deleting the Railway volume, which destroys schema + RLS and needs full re-setup. (`scripts/_reset_db.py` is a local utility, gitignored along with `*.zip` and `.import_tmp/`.)
- **Checking progress without disturbing the import:** read the script's stdout log; check the local process is advancing CPU; only query the DB with a short `SET statement_timeout` and accept it may briefly block behind the import. Don't poll the DB in a tight loop during a load.

## Startup & RLS on a giant dataset (hard-won — don't regress)

After the full dataset (1.5M `kg_node` rows) was loaded, the deployed app appeared to "lose" the dataset: `/api/health` returned 200 but `/api/datasets` timed out forever, and the UI showed "Pilih atau import dataset". Data was never lost — `dataset_catalog` had the row the whole time. The causes, in the order they were found:

- **`DATABASE_URL` must be set on the *app* service, not just Postgres.** On Railway the KGRRE service had no `DATABASE_URL`, so the pool tried to connect to nothing and hung. Fix: add `DATABASE_URL = ${{Postgres.DATABASE_URL}}` (a reference variable) on the KGRRE service so it auto-tracks Postgres credentials even after a password rotation.
- **Never connect to the DB inside `@app.on_event("startup")`.** FastAPI runs the startup hook *before* uvicorn serves requests, so a slow/hanging DB connect there blocks `/api/health` and the healthcheck never passes (Railway shows "1/1 replicas never became healthy"). `_startup()` only calls `ensure_dirs()`; schema is built lazily on the first request via `ensure_schema_once()` (called from `db_for()`/`list_datasets()`). `connect_timeout=10` is appended to the DSN in `_dsn()` so a wrong `DATABASE_URL` fails fast instead of hanging.
- **Pool must use `open=True` (synchronous).** `open=False` + `open(wait=False)` opens connections on a background thread; `pool().connection()` then still blocks indefinitely waiting for a connection if the DB is slow. `open=True` creates the pool on the first request (not at startup, so health stays fast) and `connect_timeout` bounds the wait.
- **`ensure_schema()` must NOT run `enable_rls()` in the read path — the worst regression.** `enable_rls()` runs `ALTER TABLE kg_node ENABLE/FORCE ROW LEVEL SECURITY`, which needs an **exclusive lock** on a 1.5M-row table. `list_datasets()` called `ensure_schema()` on *every* `/api/datasets` request, so each request queued a fresh `ALTER TABLE` behind older long-running `SELECT`s. `pg_stat_activity` showed **50+ `ALTER TABLE kg_node ENABLE ROW LEVEL SECURITY` backends stuck on `wait_event=Lock` for hours**, piling up without end → permanent timeout. RLS is created **once at import time** (importer.py calls `enable_rls` directly) and is already permanent (`relrowsecurity`/`relforcerowsecurity = true`, policy `dataset_isolation` on all tables). So **`ensure_schema()` now only runs `initialize()` = `CREATE TABLE IF NOT EXISTS`** (instant, no data lock), guarded by `_SCHEMA_READY` to run once per process. Never reintroduce `enable_rls()` into any read-path/per-request call.
- **Clearing a stuck-lock pile-up: restart the Postgres service**, don't `pg_terminate_backend` (interfering with shared infra). A Postgres restart drops the app's orphaned connections and Postgres rolls back their stuck transactions, releasing the locks. Data is safe (it lives on the volume).

## Heavy dashboard endpoints on a giant dataset (hard-won — don't regress)

The **Executive RU** (`/ru-summary`) and **Reliability Insight** (`/insights/reliability`) pages showed `0`/`—` for every metric and felt frozen, even though all data was present (678k `reliability_observation`, 216k `equipment`, etc.). Cause: both endpoints **recompute over the full graph on every page open**, exceeding Railway's ~30s HTTP-proxy timeout, so the request never returns and the frontend renders its zero-defaults. Measured (local, against prod DB): `reliability_insight` ≈ **38s** (7 live queries; the `equipment↔reliability_observation` joins seq-scan `kg_relationship` because RLS turns `dataset_id` into a runtime `current_setting()` that the planner won't push into the leading index column), `_readiness_association_summary` inside `ru_summary` ≈ **72s** (pulls all 216k equipment rows into Python for n-gram tag matching). Indexes already exist and are not the bottleneck — the work is just genuinely large.

- **Don't compute heavy insights inline on the request.** A 38–72s endpoint dies at the proxy before the result (or any cache) is produced, so naive caching alone never warms. The fix in `app.py`: `_cached_insight(dataset_id, key, compute, fallback)` serves a per-dataset in-process TTL cache (1h); on a cold miss it starts the `compute()` on a **daemon background thread** and immediately returns `fallback` with `{"computing": true}`. Subsequent calls return the cached full result instantly. Cache is invalidated on dataset delete (`_invalidate_insight_cache`); a process restart clears it (acceptable — dataset is static).
- **Frontend polls while `computing`.** `useWarmingData()` in `App.tsx` re-fetches every 3s until the response lacks `computing`, showing a `ComputingBanner` spinner meanwhile. The fallback shapes must match the real response (`_RELIABILITY_FALLBACK`, and the `ru_summary` fallback) so the page renders without errors during warm-up. `computing?` is optional on `RuSummary`/`ReliabilityInsight` in `types.ts`.
- **Don't "optimize" by deleting the precompute-from-`graph_analysis` reads.** `ru_summary` already correctly reads ETL-precomputed rows from `graph_analysis` (fast); only `_readiness_association_summary` is the slow live part. The page subtitle ("memakai file summary output ETL, bukan menghitung ulang graph besar") is the intended contract — keep heavy live recomputation off the page-open path.

## `/search` full-text on a giant dataset (hard-won — don't regress)

The `/search` endpoint matched `q` by casting the **entire `properties_json` jsonb to text** and `LIKE '%q%'` over all 1.5M `kg_node` rows: `lower(label||business_key||node_id||properties_json::text) LIKE %s`. That forces a full seq-scan no index can serve — measured **~15.5s** per search against the prod DB. Worse, it had **no `statement_timeout`**, so when Railway's proxy timed out and the user re-typed, each abandoned search kept scanning server-side: `pg_stat_activity` showed **9 `SELECT * FROM kg_node` backends stuck active for up to 4.5 hours**, starving all other queries (including an unrelated import). Clearing them needed a Postgres restart (don't `pg_terminate_backend` shared infra).

- **Search only the compact columns** (`label`, `business_key`, `node_id`) plus the `kg_identifier` EXISTS sub-query — **not** `properties_json::text`. Measured ~0.6s vs 15.5s with near-identical recall (41,213 vs 41,286 matches for `%pump%`), because the meaningful text already lives in `label`/`business_key`. Free-text search *inside* `properties_json` is the job of the separate Property Query feature, not `/search`.
- **Always `SET LOCAL statement_timeout` on heavy read endpoints.** `/search` sets `'20s'` so a pathological query fails fast instead of piling up as an orphaned backend. `SET LOCAL` scopes to the current transaction (the pooled connection runs statements in an implicit txn after `set_config('app.dataset_id',…)`) and resets on connection return — verified it actually cancels a slow query.
- `pg_trgm` is **not** installed and isn't needed — column search is already fast enough; don't add a trigram GIN index to the 1.5M-row table to "fix" search.

## Reliability diagnosis prompts (Graph Explore) — design contract (don't regress)

The "Reliability diagnosis prompt" panel on Graph Explore copies a role-specific LLM prompt to the clipboard; the user pastes it into an **external text LLM** (ChatGPT/Claude/Gemini) and reads the answer there — **there is no chart/markdown renderer in KGRRE for this flow.** Four buttons → four roles (`engineer`, `reliability_manager`, `maintenance_manager`, `vp`), each modelling a distinct CMRP mindset and altitude (technical → strategy/effectiveness → execution/efficiency → executive/portfolio).

- **One backend helper feeds all four.** `_reliability_engineering_signals(connection, node_id, item, properties)` in [app.py](backend/app.py) runs **per-equipment** aggregates and is called from `_readiness_context_for_node` (only when `node_type=='equipment'`). It follows the `equipment_action_queue` pattern: aggregate over `reliability_observation` / `maintenance_order` / `rkap_program` nodes via the existing `EQUIPMENT_HAS_*` edges, reading ETL-precomputed `derived_*` node properties (e.g. `derived_order_age_days`, `derived_is_open_order`, `derived_planned_cost`, `derived_status_bucket` with `WAMA`/`WASR` = wait material/spare; `derived_total_equivalent_idr_num`, `derived_is_top_risk`, `derived_is_delayed`). The numeric-cast guard (`~ '^-?[0-9]…'`) is mandatory because `mtbf`/`mttr`/cost columns are often `0`/empty. **Always `SET LOCAL statement_timeout`; never widen this into a graph-wide or RU-wide recompute on the page-open path.** The whole helper is wrapped in try/except returning `{}` so it can never break readiness-context.
- **Anti-fabrication is the core rule (don't weaken).** These are genuinely absent from the data and must stay **recommend-only / framing-only** — the LLM may *recommend* them or pose them as assurance questions, but must never claim their values: condition-monitoring tools (vibration/oil/thermography), formal Weibull β/η, OEE, ROI/RoNA, profitability, the People & Technology pillars, legal/market-image risk. MTBF/MTTR from the domain columns are mostly `0` → treat `0`/empty as "belum tercatat", never "perfectly reliable". Equipment↔history matching is **fuzzy** (`_equipment_match_tokens`) → always carry a confidence caveat.
- **Output structure is fixed: Fact Finding → Analysis → Summary** (built in `outputFormatLines()` inside `buildSharedDiagnosticContext`). **Raw numbers/units are allowed ONLY inside the Fact Finding table**; Analysis & Summary stay narrative/operational with no raw numbers, column names, or "= 0". The Fact Finding table's `Confidence Level` column is icon-only (`■■■`/`■■□`/`■□□` rating, never the words tinggi/sedang/rendah). Summary must open with a `Diagnosis/Confidence Level: [🔴/🟡/🟢] [■■■/■■□/■□□]` line, then the diagnosis paragraph, then a Confidence Level sentence (confidence sits right after the diagnosis, not after the recommendations), then a numbered (`1. 2. 3.`) recommendation list.
- **Charts are real rendered chart widgets, not tables.** Each role builder appends its own chart instruction (up to 3 bar/line/pie charts from a role-specific menu, inserted inside Fact Finding, before Analysis, never after Summary, rendered only when the underlying data is actually present — skip if 0/empty, never fabricate) worded to trigger the target LLM's native chart-rendering tool — framed as if the user had explicitly asked "buatkan saya grafik dari data ini" — and explicitly forbids any markdown/Excel-style table or ASCII art as a substitute. This replaced an earlier, weaker "(Opsional) data spec for Excel" bullet (commit `62bfb58`) that produced no visuals at all. This is a prompt-wording lever only — KGRRE can't force ChatGPT/Claude/Gemini to actually render a widget, only make the instruction as unambiguous as possible; if a target LLM still falls back to a table, that's the external LLM's behavior, not a KGRRE bug.
- **Language:** Indonesian technical prose, but **reliability/maintenance jargon may stay English** (MTBF, bad actor, root cause, P-F interval, schedule compliance, business case, asset integrity, criticality…). Only **data-structure terms are banned** (node/edge/graph/candidate relationship/column-table names/"= 0"). This distinction lives in the "Bahasa & gaya" block; don't collapse it back into a blanket "ban these terms in any language".
- All four `build*DiagnosticPrompt` builders share `buildSharedDiagnosticContext` (persona header + evidence + style rules + output format), then map their role-specific points onto the three output sections. Add new roles by extending the `DiagnosticRole` union, `buildRoleDiagnosticPrompt`, `copyPrompt`, the button row, and the `copied===role` labels together.
